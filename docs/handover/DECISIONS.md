## D-001 接手治理总原则

**日期**：2026-04-07  
**状态**：已确认  
**决策**：当前阶段优先级为：**先止血安全 → 再收口架构 → 再打通主链 → 最后扩展功能**。

**原因**：
- 当前最大风险不是功能不足，而是基础可信度不足。
- 如果继续扩功能，会放大安全、权限、流程和维护成本问题。

**影响**：
- 非主链功能默认降级优先级。
- 技术治理工作优先于新功能开发。

## D-002 与老板的协作机制

**日期**：2026-04-07  
**状态**：已确认  
**决策**：
- 产品方向、产品定位、会影响对外口径/成本/节奏的技术方向：必须先与老板商量。
- 纯技术实现、修复、重构、工程治理：负责人自行决策推进。

**原因**：
- 降低沟通成本
- 提高技术执行效率
- 避免技术细节反复打扰业务决策

**影响**：
- 日常技术事项不逐项汇报
- 只有方向变更类问题才升级沟通

## D-004 当前唯一可信主链

**日期**：2026-04-07  
**状态**：已确认  
**决策**：当前阶段所有工作优先服务于以下业务主链：

1. 企业注册
2. 企业审核
3. 获得身份/凭证
4. 求职 Agent 注册与建档
5. 企业发布职位
6. 求职侧提交申请
7. 企业推进申请状态
8. 条件满足后解锁联系方式

**原因**：
- 该主链是平台成立的最小闭环
- 其他能力在该主链未稳定前都不是优先事项

**影响**：
- 任务优先级、修复顺序、测试覆盖范围都以此为核心

---

## D-005 身份体系收口方向

**日期**：2026-04-07  
**状态**：技术建议，待落地  
**决策**：建议以 `Tenant / Principal / Credential` 作为未来统一身份主线，旧 `Agent / Enterprise / APIKey` 体系只保留兼容层，不再作为新数据入口。

**原因**：
- 当前旧新模型并存，已造成边界混乱
- 新模型更适合权限、审计、计费和多主体扩展

**影响**：
- enterprise legacy 入口应逐步下线
- 需要迁移策略、兼容策略、接口口径统一

**备注**：
- 这是当前最佳技术方向判断
- 若未来影响产品表达或对外方案，需要与老板确认后再升级为正式产品决策

---

## D-006 文档即协议，必须与实现同步

**日期**：2026-04-07  
**状态**：已确认  
**决策**：对于 AgentHire，文档不是附属品，而是协议的一部分；PRD/API/README 必须与实现同步更新。

**原因**：
- 项目面向 Agent 接入，调用方高度依赖文档自动执行
- 文档失真会直接导致集成失败

**影响**：
- 文档更新将被纳入每轮整改范围
- 废弃接口必须显式标注 deprecated
