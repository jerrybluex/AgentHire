# AgentHire 项目接手文档

更新时间：2026-04-07
当前负责人：无敌
状态：已接手 / 进入第一阶段治理

---

## 1. 项目一句话定义

AgentHire 是一个面向招聘场景的 Agent 协议/平台，目标是让求职侧 Agent、企业侧 Agent、平台管理端在统一协议和身份体系下完成注册、发现、申请流转与联系方式解锁。

---

## 2. 当前接手结论

当前项目不是从零开始，也不是单纯“收尾抛光”阶段，而是：

- 产品骨架已成型
- 前后端页面与 API 已具备雏形
- 核心主链路尚未稳定闭环
- 安全、架构一致性、测试兜底存在明显短板

总体判断：**可接手、可修复、可继续推进，但必须先治理，再扩功能。**

---

## 3. 当前最高优先级判断

### P0：必须先处理
1. 安全止血
2. 身份体系收口
3. 主链路阻断问题修复
4. 文档与代码重新对齐

### P1：一周内必须形成结果
1. 跑通唯一可信主链
2. 关闭 legacy 新数据入口
3. 补最小可用鉴权与权限边界
4. 建立可交接的问题与决策系统

### P2：一个月内完成系统治理
1. 领域边界重构
2. 状态机固化
3. 测试体系补齐
4. 前端从展示页升级为业务台

---

## 4. 当前唯一可信推进主链

后续所有技术推进优先服务于以下主链：

1. 企业注册
2. 企业审核
3. 获得身份/凭证
4. 求职 Agent 注册与建档
5. 企业发布职位
6. 求职侧提交申请
7. 企业推进申请状态
8. 满足规则后解锁联系方式

说明：
- 未服务于该主链的功能，默认降级优先级
- 非必要不继续扩展 A2A 高阶能力、复杂匹配、非核心商业化页面

## 5. 已确认的管理原则

### 与老板的协作原则
- 产品方向问题：必须先确认
- 纯技术实现、工程治理、代码修复、重构：负责人自行决策推进
- 任何会影响产品方向、节奏、成本、对外口径的技术决策：升级沟通

### 与后续接手人的交接原则
- 不允许“重新摸排一遍”
- 所有问题必须记录定位路径、影响范围、优先级、当前状态
- 所有关键决策必须记录原因、替代方案、影响面
- 文档持续更新，不能停留在一次性汇报

## 6. 当前已识别的核心风险

### A. 安全风险（高危）
- 仓库内存在敏感凭证与令牌
- 多个接口鉴权未真实落地
- 管理面、导出面、企业面存在越权风险
- 上传文件与部署暴露面不安全

### B. 架构风险（高）
- 旧身份模型与新身份模型并存
- enterprise 流程存在 legacy 与 v2 双入口
- 文档和实现语义不一致

### C. 可靠性风险（高）
- 主链路接口存在代码级阻断错误
- 上传流程状态源不一致
- 配额统计并发下可能超发
- 错误状态码与业务失败语义不一致

### D. 工程化风险（中高）
- 测试大量占位，真实保护不足
- 前端关键页面缺少自动化测试
- 多个超大文件长期提升维护成本
